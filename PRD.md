# ProspectFold — Product Requirements Document

**Product:** ProspectFold (Prospect Intelligence Engine)
**Studio:** Foxworks Studios
**Status:** Active Development
**Last Updated:** 2026-03-05

---

## 1. Overview

ProspectFold is a desktop application that turns a NAICS industry code into a complete, actionable B2B prospect intelligence package. It bridges the gap between "we sell to tech companies" and "here are the 12 specific companies in Austin hiring AI engineers right now, here's how to find the decision-maker, and here's the one-line hook that will land."

The tool is purpose-built for Foxworks Studios' sales motion: high-ticket AI engineering engagements ($15k–$100k), sold directly to technical decision-makers (CTOs, VPs of Engineering, Heads of Product) at companies that are already spending on engineering but not yet on AI.

ProspectFold does not replace a CRM. It feeds one. The output of ProspectFold is designed to flow directly into EventFold CRM's Apollo Search pipeline via a one-click clipboard bridge.

---

## 2. Problem Statement

### The current state of prospecting at Foxworks
- Prospecting is done ad-hoc: someone Googles, browses LinkedIn, and guesses at angles
- There is no shared ICP definition — each rep or founder uses their own mental model
- Apollo searches are constructed from scratch each time with no institutional memory
- No consistent qualification criteria — deals enter the pipeline without being vetted
- Context is lost between prospecting sessions; there is no "prospect intel package" to hand off

### What this costs
- Hours of research per industry vertical before a single outreach message is sent
- Low signal-to-noise in the CRM pipeline (unqualified leads clog it)
- Inconsistent messaging — different angles pitched to the same company type
- No feedback loop — there is no way to see which NAICS codes produce the best deal velocity

## 3. Target User

**Primary:** Foxworks Studios founders and business development leads
**Secondary:** Any solo founder or small team running high-ticket B2B outreach without a full SDR team

**User profile:**
- Technically literate (comfortable with API keys, JSON, developer tooling)
- Sells a high-value service or product with a long sales cycle
- Uses Apollo.io for prospecting and a CRM for pipeline management
- Does not have a dedicated research or SDR function
- Values precision over volume — would rather have 5 perfect leads than 500 generic ones

## 4. Core Concepts

### NAICS Code
The North American Industry Classification System code is the entry point for all ProspectFold intelligence. Selecting a NAICS code scopes the entire output — every angle, every search query, every qualifying criterion — to companies in that specific vertical.

### Intel Package
A structured JSON document generated by the AI engine. Contains:
- ICP summary (2-sentence ideal customer profile)
- Company types to target within the vertical
- Company sizes (employee ranges, funding stages, revenue bands)
- Qualifying criteria (observable attributes that make a company a fit)
- Buying signals (company-level triggers that indicate intent)
- Prospecting angles (named hypotheses for why each segment buys)
- Search queries for Apollo, Google, and LinkedIn
- Qualification checklist (per-company due diligence steps)
- Red flags (instant disqualifiers)
- Enrichment URLs (directories, job boards, signal sources specific to the vertical)

### Prospect Intel Payload
A small JSON blob written to the clipboard when "→ EventFold" is clicked. Contains `industry`, `titles`, `seniority`, and `keywords` — the minimum fields needed to run a live Apollo People search in EventFold CRM.

## 5. Feature Specifications

### F-01 · NAICS Industry Selector

**Description:** The primary entry point. The user selects a NAICS code to scope the intel generation run.

**Requirements:**
- Searchable dropdown with ~300+ NAICS codes grouped by sector
- Search works on code number, label, and sector name
- Suggested codes surface Foxworks' highest-priority verticals first
- Selected code is displayed as a dismissible chip (code + label)
- Changing the code clears the current result and resets state
- Selected code + label persists in the intel package output and export filenames

**Sectors covered:**
Technology, Finance & Insurance, Legal, Professional Services, Marketing & Advertising, Healthcare, Real Estate, Construction, Logistics & Transportation, HR & Staffing, Education, Hospitality, Manufacturing, Wholesale Trade, E-commerce & Retail, Media & Publishing, Energy & Cleantech, Agriculture, Mining & Resources, Arts & Entertainment, Government & Public Sector

---

### F-02 · Company Size Filter

**Description:** Optional multi-select filter that scopes the AI to companies of a specific headcount range.

**Options:** 1–10, 11–50, 51–200, 201–500, 500–1000, 1000+
**Behavior:** Selected sizes are included in the system prompt context; AI tailors qualifying criteria, signals, and search filters accordingly. Unselected = no restriction.

---

### F-03 · Additional Context Field

**Description:** Optional free-text field for user-provided context that shapes the AI output.

**Examples:**
- "We sell custom AI agents, avg deal $15k, focus on ops-heavy companies"
- "Target companies already using OpenAI API or Anthropic"
- "Avoid enterprise — Series A/B sweet spot"

**Behavior:** Appended verbatim to the system prompt. Allows the same NAICS code to produce materially different outputs depending on Foxworks' current positioning or campaign focus.

---

### F-04 · AI Intel Generation (Streaming)

**Description:** The core engine. Calls the Anthropic Messages API with a structured system prompt and streams the JSON response.

**System prompt produces:**
```
{
  summary, icp: { company_types, company_sizes, qualifying_criteria, signals },
  angles: [{ name, hypothesis, hook }],
  searches: {
    apollo: [{ label, query, filters: { industry, employee_count, keywords,
               technologies, person_titles, seniority } }],
    google: [{ label, query }],
    linkedin: [{ label, query, url_hint }]
  },
  qualification_checklist: [{ criterion, how_to_verify }],
  red_flags,
  enrichment_urls: [{ label, url, why }]
}
```

**Streaming behavior:**
- Live JSON streamed to a terminal-style display (dark background, monospace)
- Phase label updates in real time: "Profiling industry..." → "Building search queries..." → "Generating angles..."
- Spinner in the Generate button during loading
- On completion, auto-switches to the Apollo search tab

**Error handling:**
- API key validation (empty key, bad key)
- Network failure
- Malformed JSON recovery (strips ` ```json ` fences before parsing)

### F-05 · Results Display

**Description:** Structured tabbed display of the generated intel package.

**Components:**

**ICP Summary Banner** — Full-width indigo card with the 2-sentence summary at the top of results.

**Export Controls** — Two buttons above the ICP banner:
- `↓ JSON` — downloads `prospect-{naicsCode}-{date}.json` with a meta wrapper `{ version, naicsCode, naicsLabel, generated, ...result }`
- `⊞ Copy Markdown` — writes a full formatted markdown document to clipboard (summary, ICP, signals, angles, searches, checklist, red flags)

**ICP Details Grid** — Two-column grid:
- Company Types (indigo tags)
- Qualifying Criteria (green checklist)

**Buying Signals** — Two-column grid of amber diamond-prefixed signal strings

**Prospecting Angles** — Cards with angle name, hypothesis paragraph, and indigo-bordered hook quote

**Search Queries (tabbed):**
- Apollo tab — SearchCard components with live search + EventFold send
- Google tab — SearchCard components with copy query button
- LinkedIn tab — SearchCard components with URL hint

**Qualification Checklist** — Two-column table: criterion + how to verify
**Red Flags** — Red-bordered tag list
**Enrichment URLs** — Linked resource cards with "why" annotation

### F-06 · Apollo SearchCard (Live Search)

**Description:** Each Apollo search entry in the intel package is a card with an inline live-search capability.

**Card anatomy:**
- Label badge (amber)
- Query description text
- Filter chips: industry, employee count, keywords, technologies, person titles, seniority
- "Search Live" button — triggers Apollo Companies API call via Electron IPC
- "→ EventFold" button — writes Prospect Intel payload to clipboard

**Live search behavior:**
- Requires Apollo API key to be configured
- Calls `POST https://api.apollo.io/api/v1/mixed_companies/search` via Electron main process (bypasses CORS)
- Maps filter fields to Apollo API parameters:
  - `keywords` → `q_organization_keyword_tags`
  - `industry` → `organization_industries`
  - `technologies` → `currently_using_any_of_technology_uids`
  - `employee_count` → `organization_num_employees_ranges`
- Results render inline: company name, website, employee count, LinkedIn URL
- Error state shown inline if Apollo key missing or API call fails

**→ EventFold behavior:**
- Writes JSON to clipboard: `{ __prospect_intel: true, industry, titles, seniority, keywords }`
- Button flashes green "✓ Copied" for 2 seconds
- User switches to EventFold, clicks "Paste from Prospect Intel" in Apollo Search header
- EventFold fills industry, titles, seniority fields and shows confirmation toast

**Description:** Two export formats for the complete intel package.

**JSON Export (`↓ JSON`):**
```json
{
  "meta": {
    "version": 1,
    "naicsCode": "541511",
    "naicsLabel": "Custom Software Development",
    "generated": "2026-03-05T20:00:00.000Z"
  },
  "summary": "...",
  "icp": { ... },
  "angles": [ ... ],
  "searches": { ... },
  "qualification_checklist": [ ... ],
  "red_flags": [ ... ],
  "enrichment_urls": [ ... ]
}
```

**Markdown Copy (`⊞ Copy Markdown`):**
Full document with H1/H2 headings, bullet lists, and code blocks. Suitable for pasting into Notion, Linear, a pitch deck, or a Slack message. Covers all intel package sections in human-readable format.

### F-08 · History Log

**Description:** Persisted run history displayed below the input panel.

**Behavior:**
- Last 15 runs stored in `localStorage` as `prospect_history`
- Each entry: NAICS code, label, company sizes, context, timestamp, full result
- Click any entry to restore the full result (code, label, sizes, context, result tab)
- Timestamps displayed as relative time ("3 hours ago", "2 days ago")
- History survives app restarts

---

### F-09 · API Key Management

**Description:** Two API keys required; both stored in `localStorage`.

**Anthropic Key (`anthropic_key`):**
- Password input in header, right side
- Green border/text when set
- Link to `console.anthropic.com/settings/keys`
- Required for intel generation

**Apollo Key (`apollo_key`):**
- Password input in header, right side (alongside Anthropic key)
- Amber border/text when set
- Link to `app.apollo.io/#/settings/integrations/api`
- Required only for live Apollo company search (F-06)

---

### F-10 · EventFold CRM Integration (Clipboard Bridge)

**Description:** One-click bridge from ProspectFold intel to EventFold's Apollo Search form.

**ProspectFold side (this app):**
- "→ EventFold" button on each Apollo SearchCard
- Writes to clipboard: `{ __prospect_intel: true, industry, titles, seniority, keywords }`
- Visual confirmation: button flashes "✓ Copied"

**EventFold side (separate repo — `PROSPECT_INTEL_INTEGRATION.md`):**
- "Paste from Prospect Intel" button in Apollo Search page header
- Reads clipboard, validates `__prospect_intel: true` flag
- Fills `industry`, `titles`, `seniority` filter fields
- Shows indigo toast: "Fields loaded from Prospect Intelligence"
- No Rust changes, no new dependencies — single TSX file modification

## 6. Non-Features (Explicit Exclusions)

The following are intentionally out of scope for ProspectFold:

- **Contact/lead storage** — ProspectFold does not store people. It generates search parameters. Leads live in EventFold.
- **Email sending or sequencing** — Out of scope. Use EventFold + an email tool.
- **Company database** — ProspectFold does not maintain its own company records. Apollo is the source of truth for company data.
- **Team/multi-user support** — Single-user desktop app only.
- **Cloud sync** — All data is local. No backend, no accounts, no sync.
- **Mobile** — Desktop only (macOS arm64 primary target).

## 7. Technical Architecture

### Runtime
- **Framework:** Electron 33 (aarch64 macOS)
- **Frontend:** Vite 5 + React 18 (single JSX file: `prospect-crafter.jsx`)
- **Build:** electron-builder → `.dmg` (APFS, arm64)

### Process Model
```
Renderer (React/Vite)
  └─ window.electronAPI.searchApolloCompanies(filters)
       └─ ipcRenderer.invoke('apollo:companies', args)
            └─ ipcMain.handle('apollo:companies', ...)  ← Node.js HTTPS
                 └─ POST https://api.apollo.io/api/v1/mixed_companies/search

Renderer
  └─ navigator.clipboard.writeText(prospectPayload)
```

### Key files
| File | Purpose |
|------|---------|
| `prospect-crafter.jsx` | Entire frontend: state, UI, AI streaming, export |
| `electron/main.cjs` | Electron main: window, IPC handlers, Apollo HTTPS proxy |
| `electron/preload.cjs` | contextBridge: exposes `searchApolloCompanies`, `openEventFold` |
| `public/logo.png` | Foxworks fox-gear logo |
| `vite.config.js` | Vite config, base `'./'` for Electron file:// protocol |

### Apollo CORS workaround
Apollo's API does not allow browser CORS. All Apollo API calls are routed through the Electron main process (Node.js `https` module) via IPC. The renderer never calls Apollo directly.

### AI Streaming
Anthropic Messages API called from the renderer with `stream: true`. Server-sent events (SSE) are parsed via `ReadableStream` / `TextDecoder`. Accumulated JSON is parsed after stream closes.

## 9. Planned Features (Backlog)

### P1 — High priority

**Saved Intel Packages**
Allow users to name and save intel packages to a local library. Currently history is limited to 15 LRU entries with no names.

**Signal Alerting**
Monitor a list of saved NAICS codes for new companies matching ICP criteria. Requires a lightweight scheduler and a persistent background check against Apollo or a web scraping source.

**Multi-NAICS Campaign View**
Side-by-side comparison of ICP summaries and angles across 2–3 NAICS codes. Useful for scoping a campaign that spans verticals (e.g., "fintech + insurtech + proptech").

**ProspectFold → EventFold Deal Pre-fill**
Extend the clipboard bridge to also pass company name, company size, and angle name, enabling EventFold to pre-populate a new Deal record — not just the Apollo Search form.

### P2 — Medium priority

**Custom Signal Sources**
Allow users to add their own enrichment sources (Crunchbase saved searches, job board RSS feeds, LinkedIn alerts). ProspectFold checks these on open and surfaces new matches.

**Angle Scoring**
After running a few deals from a given NAICS, allow the user to mark which angles converted. Surface winning angles more prominently in future runs for the same NAICS.

**Template Library**
Save additional context snippets as reusable templates. E.g., "Foxworks AI Agent pitch" or "Ops automation focus" — one click to pre-fill the context field.

**Google Search Tab Enhancements**
Auto-generate clickable Google search links and LinkedIn Sales Navigator deep links from the generated query strings.

### P3 — Lower priority

**PDF Export**
Formatted PDF version of the intel package suitable for sharing with a client or partner.

**Webhook Output**
POST the intel package JSON to a configurable webhook URL (Zapier, Make, n8n) on generation complete. Enables downstream automations.

**OpenAI / Local LLM Support**
Allow swapping the Anthropic backend for OpenAI or a local Ollama model. Low priority since Claude's structured output quality is the core value driver.

## 11. Open Questions

1. **Multi-user:** If a second Foxworks team member needs ProspectFold, is the single-user model acceptable or do we need shared history/saved packages?
2. **Apollo rate limits:** The live search feature calls Apollo's API on every "Search Live" click. What is the rate limit and do we need request debouncing or a cache layer?
3. **Intel freshness:** How often should a NAICS intel package be re-generated? Should there be a staleness indicator ("Generated 30 days ago")?
4. **EventFold coupling:** As the EventFold CRM evolves, should ProspectFold track EventFold's schema changes, or should the clipboard payload be versioned?
5. **Repo structure:** Does ProspectFold stay a standalone Electron app long-term, or does it eventually become a view within EventFold itself?
